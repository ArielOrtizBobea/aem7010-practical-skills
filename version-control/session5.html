Session 5: GitHub & Collaboration

Working with remotes and teams

Quick Recap

In Session 4 we learned the core local Git workflow:

  • git init: create a repository
  • git add / git commit: stage and save snapshots
  • git log / git diff: view history and changes
  • .gitignore: exclude files from tracking
  • git restore / git reset: undo mistakes

Today we take those skills online.

What Is GitHub?

GitHub is a cloud hosting service for Git repositories. It provides:

  • Backup: your code lives in the cloud, not just on your laptop
  • Collaboration: co-authors can contribute to the same project
  • Sharing: publish replication packages and make research transparent
Note

GitHub is free for public and private repositories. You can also apply for GitHub Education benefits as a student or faculty member.

Keep Your Email Private

Every commit you push to a public GitHub repository shows the name and email you configured back in Session 4. If you used a personal email, it is now visible to anyone browsing the repo. GitHub offers a stable noreply address tied to your account that you can use instead.

  1. Go to github.com/settings/emails.
  2. Under Primary email address, check the box for Keep my email addresses private.
  3. GitHub shows a noreply address shaped like 12345678+yourusername@users.noreply.github.com. Copy it.
  4. Update your local Git configuration:
git config --global user.email "12345678+yourusername@users.noreply.github.com"

Replace the address with the one shown on your GitHub settings page.

Tip

This only affects commits made after you change the setting. Past commits keep whatever email they were created with. Rewriting history to replace old emails is possible but advanced. See GitHub’s documentation if you need it.

Connect to GitHub: SSH Keys

SSH keys let your computer authenticate with GitHub without typing a password every time.

1. Generate a key pair

ssh-keygen -t ed25519 -C "you@email.com"

Press Enter to accept the default file location. You can set a passphrase or leave it empty.

2. Start the SSH agent and add your key

eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

3. Copy the public key

cat ~/.ssh/id_ed25519.pub

Copy the entire output.

4. Add the key to GitHub

  1. Go to github.com → click your profile picture → Settings
  2. In the left sidebar, click SSH and GPG keys
  3. Click New SSH key
  4. Paste your key and give it a title (e.g., “My Laptop”)
  5. Click Add SSH key

5. Test the connection

ssh -T git@github.com

You should see: Hi username! You've successfully authenticated...

Tip

Windows users: run these commands in Git Bash, not in PowerShell or Command Prompt.

Remotes: Push and Pull

A remote is a copy of your repository stored on a server (like GitHub). The standard remote name is origin.

The relationship

Your Computer  ←— git pull —→  GitHub (origin)
               —— git push ——→
  • git push uploads your commits to GitHub
  • git pull downloads new commits from GitHub

Link your local repo to GitHub

First, create a new repository on GitHub:

  1. Go to github.com/new
  2. Name it (e.g., wage-analysis)
  3. Do NOT initialize with a README (your local repo already has commits)
  4. Click Create repository

GitHub will show you setup commands. Use the SSH option:

git remote add origin git@github.com:yourusername/wage-analysis.git

Push your commits

git push -u origin main

The -u flag sets origin main as the default upstream, so future pushes only need git push.

Pull changes

If someone else pushed commits (or you edited a file on GitHub), download them:

git pull origin main

Cloning a Repository

git clone downloads an entire repository (with full history) from GitHub to your computer:

git clone git@github.com:username/wage-analysis.git
cd wage-analysis

After cloning, the remote origin is already configured. You can immediately push and pull.

Tip

Cloning is how you start working on a collaborator’s project. You get the full history, all branches, and a ready-to-go remote connection.

Exercise 3: Push to GitHub

Time: ~15 minutes

  1. Create a repository on GitHub. Go to github.com/new, name it wage-analysis, and click Create (do NOT initialize with README).

  2. Add the remote to your local wage-analysis project:

cd wage-analysis
git remote add origin git@github.com:YOURUSERNAME/wage-analysis.git

Replace YOURUSERNAME with your GitHub username.

  1. Push your commits:
git push -u origin main

  1. Verify. Go to your repository page on GitHub and refresh. You should see your files and commit history.

  2. Test pulling. On GitHub, click on clean_data.R → pencil icon → add a comment line at the top → Commit. Then pull locally:

git pull origin main

Open the file to confirm the edit is there.

Branches

A branch is a parallel version of your code. You can experiment on a branch without affecting the main version. When the work is ready, you merge it back.

Why branches?

  • Try a new specification without breaking working code
  • Each co-author works on their own branch
  • Keep main clean and functional at all times

Create and switch to a branch

git branch add-iv-analysis
git checkout add-iv-analysis

Or, as a shortcut:

git checkout -b add-iv-analysis

Work on the branch

Make your edits, stage, and commit as usual. These commits only exist on the branch.

# Create a new file
echo '# IV regression using 2SLS' > iv_analysis.R
git add iv_analysis.R
git commit -m "Add IV analysis script"

Switch back to main

git checkout main

Notice that iv_analysis.R disappears. It only exists on the other branch. Your working directory reflects whichever branch you are on.

Merge the branch

When you are satisfied with the work on your branch:

git checkout main
git merge add-iv-analysis

Now iv_analysis.R appears on main and all the branch’s commits are part of main’s history.

Delete the branch (optional)

After merging, you can clean up:

git branch -d add-iv-analysis

Pull Requests

A pull request (PR) is GitHub’s way of proposing changes. Instead of merging locally, you push a branch to GitHub and ask for it to be reviewed before merging.

The pull request workflow

  1. Create a branch and make your commits
  2. Push the branch to GitHub:
git push -u origin add-iv-analysis
  1. Open a PR on GitHub. Go to your repository → you will see a banner saying “add-iv-analysis had recent pushes” → click Compare & pull request
  2. Write a description explaining what you changed and why
  3. Review. Your co-author reads the changes, leaves comments, and approves
  4. Merge. Click Merge pull request on GitHub
Note

Pull requests are the standard collaboration workflow in both industry and academic research. They create a written record of what changed and why, which is valuable for reproducibility.

Handling Merge Conflicts

A merge conflict happens when two people edit the same line in the same file. Git marks the conflict and asks you to choose.

What a conflict looks like

<<<<<<< HEAD
lm(wage ~ educ + exper, data = df)
=======
lm(log(wage) ~ educ + exper + tenure, data = df)
>>>>>>> add-iv-analysis
  • The top section (HEAD) is your current branch’s version
  • The bottom section is the incoming branch’s version

How to resolve

  1. Open the file in your editor
  2. Decide which version to keep (or combine them into something new)
  3. Delete the <<<<<<<, =======, and >>>>>>> markers
  4. Stage and commit the resolved file:
git add clean_data.R
git commit -m "Resolve merge conflict in regression specification"
Tip

Merge conflicts are normal, not dangerous. They happen whenever two people edit the same line. The fix is always: open the file, choose the right version, remove the markers, commit.

Exercise 4: Branches and Pull Requests

Time: ~20 minutes. Work with a partner.

Setup

  1. Person A: go to your wage-analysis repo on GitHub → Settings → Collaborators → Add your partner’s GitHub username.
  2. Person B: accept the invitation (check your email or GitHub notifications), then clone Person A’s repo:
git clone git@github.com:PERSONA_USERNAME/wage-analysis.git
cd wage-analysis

Create branches and PRs

  1. Both partners: create a branch with your name, add a new .R file, commit, and push:
git checkout -b yourname-feature

Create a file (e.g., descriptive_stats.R or robustness_check.R) with a few lines of R code.

git add your_file.R
git commit -m "Add descriptive statistics"
git push -u origin yourname-feature

  1. Both partners: go to GitHub and open a pull request from your branch into main.

  2. Review each other’s PR. Click on “Files changed” to see the diff. Leave a comment (e.g., “Looks good!” or “Maybe add a header comment?”). Then click Merge pull request.

  3. Pull the merged changes locally:

git checkout main
git pull origin main

Both files should now be on main.

Tags for Reproducibility

Tags mark a specific point in history. They are the mechanism for recording the code that produced a given version of your paper: the first submission, the R&R revision, the final accepted manuscript, the posted replication package. Years later you can return to any of those versions in one command.

A typical research project has a tag for each submission milestone:

v0.1-first-submission       (code used in the initial JPE submission)
v0.2-first-RR               (code for the first round of R&R)
v0.3-second-RR              (code for the second round)
v1.0-accepted               (final version in the published paper)
v1.0-replication-package    (the archive you posted on AEA/Zenodo/ICPSR)

Create a tag

git tag -a v0.1-first-submission -m "Code for initial JPE submission"

Push tags to GitHub

git push --tags

Check out a tag later

git checkout v1.0

This puts you in “detached HEAD” state. You can look around and run code, but any new commits won’t belong to a branch. To go back:

git checkout main

GitHub Releases

On GitHub, tags appear under the Releases tab. You can attach files (data documentation, README) to a release, making it a self-contained replication package.

Important

Many journals now require replication packages. A tagged GitHub release with a README is a clean, standard way to comply.

Mark reproducible versions git tag
